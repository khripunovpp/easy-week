## 💡 Идеи / бэклог

### Ближайшие (обсуждено, приоритет)
- **Сравнение рецептов разными моделями** (#21, в работе) — кнопка на странице блюда: показать
  варианты рецепта от разных моделей бок о бок (diff ингредиентов/шагов). Можно догенерить на Cloudflare.
- **Избранное рецептов и планов + фильтр на «Планах»** (#17) — блюдо/план в избранном; на странице
  «Планы» фильтр показывает планы, содержащие избранные блюда.
- **Мягкое удаление сообщений в чате** (#3) — кнопка «удалить» скрывает сообщение (курсив «Удалено»),
  но физически оно остаётся; в контекст модели такие сообщения НЕ попадают (ни user, ни assistant).
- **Время готовки: активное vs пассивное** (#12) — делить «у плиты» и пассивное (духовка/тушение);
  учитывать параллельность заготовки (говядина 3 ч в духовке ≠ 3 ч ручной работы).
- **Сезонность/цена** (#11) — мягкий фактор подбора блюд.
- **Кэш детерминированных рецептов между планами** (#49) — одинаковое блюдо не генерить заново,
  переиспользовать готовый вариант.

### Заморозка/хранение (киллер-домен)
- **Трекер морозилки** — что заморожено, когда, срок, напоминания «съесть до …».
- **Печать этикеток** на контейнеры (блюдо + дата + срок), опц. QR на рецепт.
- **Календарь разморозки** — «вечером достань X на завтра» (+ ICS-экспорт).
- Учёт объёма морозилки / числа контейнеров; блок «как порционировать и вакуумировать».

### Рецепты
- КБЖУ/калории на порцию; интерактивный чек-лист шагов с галочками; таймеры в шагах.
- Замены ингредиентов («нет сметаны — чем заменить»); пересчёт под другое число порций на лету.
- Режим «параллельная готовка» — слить шаги нескольких блюд в один тайм-план заготовки.

### Список покупок
- Отметки «куплено» с **синком между устройствами** (серверное хранение — сейчас только localStorage).
- Сортировка по отделам магазина; объединение закупки по нескольким планам; учёт кладовки; оценка стоимости.

### Планирование
- Бюджет плана (оценка стоимости); баланс белков/овощей/КБЖУ; «приготовь из того, что есть».
- Явно избегать блюд из недавних планов + показывать «недавно было»; масштаб плана под число едоков.

### Персонализация / дом
- Аллергии/диеты (без глютена/лактозы, кето, халяль); профили едоков; обучение по «зашло/не зашло».
- Лёгкая идентификация «кто это» (без полноценной авторизации); совместное редактирование плана; голосование за блюда.

### PWA / кухня
- Режим «на кухне»: крупный шрифт, не гаснет экран (Wake Lock).
- Экран/виджет «сегодня готовим и что разморозить»; свайпы на блюде (заменить/убрать).

### Модели / ops
- Grafana-дашборд качества: ok/false по моделям, `duration_ms`, токены/план; авто-алерт при росте ошибок.
- Голосовой ввод сообщения (Web Speech API); undo последней правки плана.

### Прочее
- **Общие компоненты плана** — бульон/соус-база, который готовится отдельно и используется в
  нескольких блюдах → план из 5 блюд, но фактически 6 рецептов (и в списке покупок ингредиенты
  базы учитываются один раз). (Базовая доработка плана в чате — добавить/убрать/заменить — уже есть.)
- **Tool calling для модели** (на потом) — дать модели инструменты искать по существующим
  планам (по блюдам/ингредиентам/неделям), опираться на реальную историю. Нужно ограничить
  контекст, чтобы не скармливать все планы целиком. Пока не приоритет — лежит до лучших времён.
- **Перевести lasagna/my-home на прод-сборки** — их dev-серверы (ng serve, ts-node)
  как systemd-сервисы едят ~2 ГБ RAM на Пае; собрать статику + nginx / скомпилировать бэки.

## ✅ Сделано
- **Экономия токенов на правках** — из edit-промпта убран полный свод правил заморозки
  (~1000 ток/правку на Claude/Gemini, где нет кэша): шаг правки только выбирает функцию, а
  заморозку применяет сама генерация рецепта/плана. Осталось короткое напоминание.
- **Обогащённое логирование AI** — в JSONL: `ok`/`error`/`attempt`, `duration_ms`, корреляция
  `conversation_id`/`plan_id`/`dish_id`/`endpoint`/`action` (contextvars) + счётчик ошибок.
- **Дедуп генерации рецепта** — single-flight на бэке по (plan_id, dish_id, action, model) +
  дизейбл кнопок на фронте: параллельные одинаковые запросы не плодят вызовы модели.
- **Варианты рецепта по моделям** — блюдо хранит по варианту рецепта на каждую модель
  (deepseek/gemini/claude/cloudflare) с переключением; бейдж «🤖 модель1 | модель2 | ⟳» под
  названием, ⟳ — попап выбора ещё не сгенеренной модели, одной моделью повторно не генерим.
  Смена активного варианта авто-пересобирает список покупок. Закрыло «generate_details → лучшая модель».
- **Доработка плана в чате (базовая)** — добавить/убрать/заменить блюдо и пересобрать меню
  по просьбе (tool calling / structured actions), правка создаёт новую версию плана.
- **Правка плана → новая карточка внизу чата** — после реплики «Готово…» новая версия плана
  уезжает вниз ленты, прошлая сворачивается как отклонённая, автоскролл к новой.
- **Отклонённый план в чате** — компактно: только название, статус и ссылка «Посмотреть план».
- **Модель чата = модель последнего плана** — при «Продолжить обсуждение» модель чата берётся
  из провайдера последнего плана (а не глобального дефолта профиля).
- **Markdown в репликах бота** — ответы моделей рендерятся из markdown (жирный/списки/код/ссылки).
- **Кнопка «вниз» в чате** — когда лента прокручена не до конца.
- **PDF одной кнопкой + шэр файлом** — «⬆ PDF»; по HTTPS уходит сам файл (Web Share), по LAN-HTTP
  открывается настоящий PDF сервера (без blob:-ссылки, которую iOS слал вместо файла).
- **Ленивая генерация деталей (экономия токенов)** — план отдаёт только «шапки» блюд;
  ингредиенты+шаги+советы генерятся DeepSeek при открытии блюда (и добэкфиллятся для покупок/PDF),
  кэшируются. Output плана упал ~70%. Закрыло правило «generate_details → DeepSeek».
- **Мониторинг** (Prometheus + Grafana + Loki + Promtail): `/metrics` бэкенда
  (токены, кэш, HTTP), логи (journald + AI-JSONL) в Loki, дашборд в Grafana.
  Локально — Docker (`docker/`, профиль `monitoring`), на Пае — native (`install-pi.sh`).
- **Полное логирование AI** — консоль + файл-за-день JSONL (промпт/ответ/usage/кэш).
- **Удаление планов** с подтверждением (модалка, необратимо) + `.btn-danger`.
- **Общее время готовки** плана — на вью плана и в списке.
- **Отмеченные покупки** уезжают вниз своей категории.
- **Убраны спиннеры** — единый кулинарный лоадер `ew-cooking` везде.
- **Единый лейаут по гайдбуку** — все экраны (Главная, Чат, Планы, План, Покупки, Печать, Блюдо)
  на каркасе `.page`: единый ритм, шапки одной высоты, фикс/не-фикс совпадают. Раздел в GUIDEBOOK.md.
- **PDF плана** — бэкенд собирает PDF (fpdf2 + DejaVu, надёжная кириллица); тумблеры
  Рецепты/Покупки с чекбоксами на странице печати. (Шэр файлом — см. пункт выше.)
- **Секции на «Планах»** — Принятые (последние 3 + «Загрузить ещё» +10) и История (отклонённые+черновики).
- **Даты на следующую неделю** — план всегда на ближайшую неделю (готовка на выходных перед ней).
- **Потоковая генерация плана** (SSE): блюда прилетают по одному, лоадер внутри карточки,
  автоскролл, кнопки только по готовности. Тот же вызов DeepSeek — токенов не больше.
- **Потоковый список покупок** (SSE) из детерминированной базы — мгновенно и точно
  (модель в потоке теряла числа, поэтому убрана из этого пути).
- **Деплой на Pi** (192.168.1.230:8080) + репозиторий на GitHub (khripunovpp/easy-week);
  Пай теперь тянет обновления через git (deploy/update.sh).
- Кулинарный лоадер (кастрюлька с паром) + пул ~100 фраз вместо спиннеров.
- Метки в логах: какой запрос уходит в какой API (провайдер · модель · назначение).
- DeepSeek для плана + короткие шаги; развёрнутые инструкции и список — Cloudflare.
- Список покупок в граммах у источника (без хаоса единиц).
- Локальная разработка в Docker с живыми логами; SW погашен в dev-сборке.
- Экран приветствия, выбор кол-ва блюд, фейд-блюр шапок/футеров, вью плана.

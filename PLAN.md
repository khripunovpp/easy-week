# План: «Easy Week» — домашний чат-планировщик рецептов на неделю

## Context

Домашнее приложение, которое **в диалоге** подбирает блюда на неделю и для каждого выдаёт: время
готовки, способ хранения (упор на **вакуумирование + заморозку**), советы, **пошаговые инструкции**,
список ингредиентов и итоговый **список покупок**.

Формат под быт пользователя (meal-prep: наготовить впрок, завакуумировать, заморозить):
- **PWA** (ставится на телефон/десктоп), работает на домашней **Raspberry Pi**.
- Бэкенд — **FastAPI** (Python), фронт — **Angular**. Генерация — **Cloudflare Workers AI** (у пользователя
  есть аккаунт), REST API со **structured JSON output**.
- Цель сейчас — **MVP**, дальше итерации.

**Ключевые требования к взаимодействию:**
- **Чатовый интерфейс.** Пользователь общается диалогом; каждый ответ модели — **JSON**, из которого фронт
  рендерит страницу (список рецептов, шаги, тайминги, хранение, список покупок). План дорабатывается
  последующими сообщениями.
- **Статусы плана: `accepted` / `rejected`** (+ `draft` пока в работе). Новая генерация **опирается на
  принятые планы прошлых недель** (не повторять недавние блюда, учитывать вкусы), отклонённые — как
  негативный сигнал.
- **Экспорт в PDF** — по частям (рецепты / список покупок / советы) **или всё вместе**.
- **Экспорт в JSON** нужного формата (для внешнего приложения подсчёта себестоимости; формат — позже).
- **Экономия токенов — обязательное требование** на всех уровнях.

Папка пустая, git инициализирован (ветка `main`). Greenfield-проект.

## Архитектура (MVP)

Монорепозиторий:
```
easy-week/
  backend/        # FastAPI + SQLite
  frontend/       # Angular PWA (чат + рендер страниц)
  deploy/         # docker-compose / инструкции для Raspberry Pi
  README.md
  .gitignore
```

**Поток данных (чат):**
1. Пользователь пишет сообщение в чат (напр. «составь неделю на 5 ужинов, без свинины, порции на 2»).
2. Backend собирает **компактный** контекст: системный промпт + краткое «ядро» предпочтений + сжатая
   выжимка принятых планов прошлых недель + текущий черновик (если есть) + новое сообщение.
3. Зовёт Cloudflare Workers AI с **json_schema** → получает структурированный JSON (план/правка/уточнение).
4. Backend сохраняет ход диалога и черновик плана в SQLite, агрегирует ингредиенты в список покупок.
5. Фронт рендерит JSON как страницу: карточки блюд, шаги, хранение, список покупок.
6. Пользователь **принимает** (`accepted`) или **отклоняет** (`rejected`) план. Принятый влияет на будущие недели.

## Backend — FastAPI (`backend/`)

Стек: `fastapi`, `uvicorn`, `pydantic v2`, `sqlmodel` (SQLite), `httpx`, `python-dotenv`.

```
backend/
  app/
    main.py            # FastAPI, CORS, монтирование статики фронта
    config.py          # env: CF_ACCOUNT_ID, CF_API_TOKEN, CF_MODEL, CF_MODEL_SMALL, DB_PATH
    db.py              # engine + сессии SQLModel
    models.py          # Conversation, Message, MealPlan(status), Dish
    schemas.py         # Pydantic-схемы + JSON-схемы ответов AI (план / уточнение)
    ai/
      cloudflare.py    # клиент Workers AI: POST /accounts/{id}/ai/run/{model}, retry+timeout
      prompt.py        # сборка КОМПАКТНОГО промпта + json_schema
      context.py       # выбор минимального контекста (см. «Оптимизация токенов»)
      planner.py       # chat_turn(conversation_id, user_msg) -> AI JSON
    services/
      shopping.py      # агрегация ингредиентов -> список покупок (merge по name+unit)
      export_json.py   # маппер во внешний JSON (себестоимость)
      export_pdf.py    # рендер частей плана -> HTML для печати/PDF
      history.py       # выжимка принятых планов прошлых недель (компактная)
    routers/
      chat.py          # POST /api/chat  (главный эндпоинт диалога)
      plans.py         # GET /api/plans, GET /api/plans/{id}, POST /api/plans/{id}/status
      shopping.py      # GET /api/plans/{id}/shopping-list
      export.py        # export JSON / PDF
  requirements.txt
  .env.example
```

**Модель данных (SQLModel):**
- `Conversation`: id, created_at, title.
- `Message`: id, conversation_id(FK), role (`user`/`assistant`), content_json, created_at.
  Для ассистента храним именно JSON-ответ (из него рендерится страница и собирается план).
- `MealPlan`: id, conversation_id(FK), week_label, **status** (`draft`/`accepted`/`rejected`),
  created_at, decided_at.
- `Dish`: id, plan_id(FK), name, servings, prep_time_min, cook_time_min, storage_method,
  shelf_life_days, tips_json, steps_json, ingredients_json (list[{name, qty, unit}]).
- Список покупок вычисляется из блюд плана (не отдельная таблица в MVP).

**Статусы и влияние на генерацию:**
- `POST /api/plans/{id}/status` со значением `accepted`/`rejected`.
- `services/history.py` формирует **короткую** выжимку по последним N принятым планам: список названий блюд
  (чтобы не повторять) + принятые предпочтения. Отклонённые планы — краткий список «не предлагать похожее».
  Это подмешивается в промпт в максимально сжатом виде (только названия/теги, не полные рецепты).

**Cloudflare Workers AI (`ai/cloudflare.py`):**
- `POST https://api.cloudflare.com/client/v4/accounts/{CF_ACCOUNT_ID}/ai/run/{model}`,
  заголовок `Authorization: Bearer {CF_API_TOKEN}`.
- Основная модель `CF_MODEL` = `@cf/meta/llama-3.3-70b-instruct-fp8-fast` (многоязычная, инструктивная).
  Отдельная **лёгкая** `CF_MODEL_SMALL` (напр. `@cf/meta/llama-3.2-3b-instruct`) для дешёвых задач
  (уточняющие вопросы, короткие правки) — экономия токенов/времени на Pi.
- **`response_format` c `json_schema`** — модель возвращает строго структурированный ответ (без парсинга прозы).
  Промпт на русском, метрические единицы, реалистичные тайминги, обязательный блок «вакуум+заморозка+срок» у блюда.

**Эндпоинты (MVP):**
- `POST /api/chat` — тело `{conversation_id?, message}`; ход диалога, возвращает JSON-ответ ассистента
  (тип: план / уточнение / правка) + обновлённый черновик плана.
- `GET /api/plans`, `GET /api/plans/{id}`, `POST /api/plans/{id}/status`.
- `GET /api/plans/{id}/shopping-list`.
- `GET /api/plans/{id}/export.json` и `/api/dishes/{id}/export.json` — выгрузка для себестоимости.
- `GET /api/plans/{id}/export.pdf?parts=recipes,shopping,tips|all` — PDF по частям или целиком.

## Оптимизация токенов (обязательное требование)

Закладываем с самого начала, тестируем на реальных диалогах:
1. **Не слать всю историю.** В контекст идёт: сжатый системный промпт + «ядро предпочтений» + текущий
   черновик плана (по id, а не переспрашивание) + последние 1–2 реплики. Старое — свёрнуто в короткое резюме.
2. **Резюмирование диалога.** При росте истории — заменять старые сообщения на 1–2 строки summary
   (можно лёгкой моделью `CF_MODEL_SMALL`).
3. **Инкрементальная генерация.** Сначала — компактный список блюд (названия, тайминги, теги хранения).
   Полные шаги/советы для конкретного блюда генерим **по запросу** (клик по карточке), а не всё сразу.
4. **Кэш блюд.** Уже сгенерированные блюда переиспользуем (по названию/хешу параметров), не перегенерируем.
5. **Строгая JSON-схема + лимиты.** Ограничивать `max_tokens`, просить краткие шаги; JSON вместо прозы.
6. **Маленькая модель для мелочей.** Уточняющие вопросы и мелкие правки — на `CF_MODEL_SMALL`.
7. **История прошлых недель — только теги/названия**, не полные рецепты (см. `history.py`).
8. **Логировать расход токенов** ответа Cloudflare (в лог/в поле сообщения), чтобы видеть эффект оптимизаций.

## Экспорт

**JSON для себестоимости (`services/export_json.py`):**
- Точный формат пользователь даст позже → изолируем в **маппере** «внутренняя модель → внешний формат»,
  меняется в одном месте. На ингредиенте держим `name`, `qty`, `unit` (нормализованные единицы), при
  необходимости `note`/`category`. Эндпоинт отдаёт `application/json` (+ опция скачивания файлом).
- **TODO:** дождаться точной JSON-схемы от пользователя (поля, вложенность, единицы, план целиком vs. один рецепт).

**PDF (`services/export_pdf.py`):**
- **По частям** (рецепты / список покупок / советы) **или всё вместе** — параметр `parts`.
- **Подход для Pi (лёгкий):** рендерим HTML-шаблон нужных частей и печатаем в PDF **на стороне браузера**
  (`window.print()` + `@media print` CSS) — ноль тяжёлых зависимостей на Pi, работает оффлайн.
  Бэкенд отдаёт готовую к печати HTML-страницу (`/export.pdf` как HTML-препринт или прямой JSON→шаблон на фронте).
- Альтернатива (если нужен серверный PDF-файл): `weasyprint`/`reportlab` — но на Pi тяжелее, оставляем на потом.

## Frontend — Angular PWA (`frontend/`)

- Angular (standalone components) + `@angular/pwa` (service worker, манифест, оффлайн-кэш GET-планов).
- **Чат — главный экран:** лента сообщений; ответ ассистента = JSON → рендерится как **богатая страница**
  (карточки блюд, тайминги, бейджи хранения, шаги, список покупок), а не просто текст.
- Экраны/блоки MVP:
  1. **Чат** — ввод сообщений, стрим ответа, кнопки-подсказки («на 5 ужинов», «без свинины»).
  2. **Страница плана** (из JSON ответа) — карточки блюд + кнопки **Принять / Отклонить** (меняют статус).
  3. **Блюдо** — пошаговые инструкции, ингредиенты, советы по вакууму/заморозке/разморозке (грузятся по клику — экономия токенов).
  4. **Список покупок** — агрегированный, чекбоксы (состояние в localStorage).
  5. **История** — прошлые планы с их статусами (accepted/rejected).
  - Кнопки экспорта: **Скачать/печать PDF** (выбор частей) и **Скачать/копировать JSON** (для себестоимости).
- Mobile-first (кухня, телефон): крупные шаги, удобно листать.

## Деплой на Raspberry Pi (`deploy/`)

- Pi (ARM64). По умолчанию **вариант A**:
  - **A (просто):** `ng build` → статику отдаёт сам FastAPI (`StaticFiles`), один процесс `uvicorn`,
    запуск через `systemd`. SQLite-файл в `deploy/data/`.
  - **B (docker):** `docker-compose`, один контейнер (uvicorn + статика), том для SQLite.
- **HTTPS для PWA:** через **Cloudflare Tunnel** (аккаунт уже есть) — бесплатный HTTPS + доступ без проброса портов.
  Альтернатива для LAN: Caddy + локальный сертификат.
- Секреты (`CF_ACCOUNT_ID`, `CF_API_TOKEN`) — в `.env` на Pi, не в git.

## Порядок реализации (итерации MVP) — frontend-first

Правило: **всегда начинаем с фронта**, чтобы сразу видеть визуал. Бэкенд сначала мокаем (статичный JSON
нужной схемы), потом заменяем на реальные вызовы Cloudflare. Разработку начинаем **только по команде пользователя**.

1. Скелет репо (`frontend/`, `backend/`, `.gitignore`, `README.md`). Первый commit.
2. **Frontend:** Angular PWA-скелет + экран **чата**. Ответы ассистента берём из **мок-JSON** (фиксированный файл нужной схемы) → сразу видим рендер. Дизайн — по `frontend-design`.
3. **Frontend:** рендер JSON→страница плана (карточки блюд, тайминги, бейджи хранения) + кнопки **Принять/Отклонить**.
4. **Frontend:** блюдо (ленивая загрузка шагов), список покупок (чекбоксы), история, кнопки экспорта (пока по моку).
5. **Backend:** конфиг + `cloudflare.py` + `prompt.py` + `planner.chat_turn` (реальный JSON вместо мока). Проверить curl-ом.
6. **Backend:** модели (Conversation/Message/MealPlan+status/Dish) + роутеры `chat`, `plans`, `shopping`. Прогон через `/docs`.
7. **Backend:** `history.py` (учёт принятых планов) + оптимизация токенов (`context.py`) + логирование расхода токенов.
8. Экспорт: PDF по частям (print CSS) + JSON-маппер (после получения формата от пользователя).
9. Деплой на Pi (вариант A) + Cloudflare Tunnel. Проверка установки PWA с телефона.

## Рекомендованные скиллы для эффективной работы

- **`/init`** — сгенерировать `CLAUDE.md` (структура, команды запуска backend/frontend) после скелета.
- **`/run`** — запускать backend (uvicorn) и frontend (`ng serve`), снимать скриншоты PWA.
- **`/verify`** — end-to-end сценарий (чат → план → принять → список покупок → экспорт) после нетривиальных изменений.
- **`/code-review`** — ревью диффа перед коммитами (особенно AI-клиент, оптимизация токенов, агрегация).
- **`/frontend-design`** — аккуратный, не «шаблонный» вид Angular-чата/страниц (mobile-first).
- **`/find-skills`** — поискать/поставить скиллы под FastAPI и Angular (в наборе есть только неподходящий `nestjs-best-practices`).
- **`/claude-api`** — как референс; помнить, что генерация — через Cloudflare Workers AI, не Claude API.

Не используем: `nestjs-best-practices` (у нас FastAPI), Figma-скиллы, `dataviz`.

## Verification (как проверить)

- **Чат/AI:** `POST /api/chat` с сообщением → валидный JSON: N блюд, у каждого `cook_time_min`,
  `storage_method` (вакуум+заморозка), `shelf_life_days`, `steps[]`, `ingredients[]`.
- **Статусы:** `POST /api/plans/{id}/status accepted` → следующий `POST /api/chat` не повторяет недавно
  принятые блюда (видно в промпт-контексте/ответе).
- **Токены:** сравнить расход на «полный план сразу» vs «инкрементально + резюме истории» — в логах должно быть меньше.
- **Список покупок:** одинаковые ингредиенты из разных блюд просуммированы.
- **Экспорт:** PDF по частям (`parts=recipes` / `shopping` / `all`) печатается корректно; JSON-выгрузка соответствует (будущей) схеме.
- **Frontend:** `ng serve` → диалог, рендер страницы плана, Принять/Отклонить, ленивые шаги, экспорт; проверить установку PWA и оффлайн-просмотр.
- **Pi:** после деплоя открыть по HTTPS (Cloudflare Tunnel) с телефона, установить как PWA.
